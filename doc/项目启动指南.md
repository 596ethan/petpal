# PetPal 项目启动指南

本文档适用于本机工作区：

```text
D:\HongMengprogram\20260317
```

项目由三部分组成：

| 模块 | 路径 | 作用 |
| --- | --- | --- |
| 手机端 | `Cutepetpost` | HarmonyOS phone client，当前主交付物 |
| 后端 | `petpal-server` | Spring Boot API，提供登录、预约、宠物档案、社区、文件代理 |
| 管理端 | `petpal-admin` | 轻量 HTML 管理页，用于服务商和预约状态管理 |
| 本地依赖 | `deploy/docker-compose.yml` | Redis、MinIO；业务数据库使用本机 MySQL |

当前默认后端 HTTP 端口是 `18080`，不是 `8080`。本机 Windows 可能保留 `8000-8099` 端口段，`8080` 容易不可用。

## 1. 环境准备

### 1.1 必需软件

请确认已安装：

- Docker Desktop
- JDK 17 或更高版本
- Maven
- DevEco Studio
- HarmonyOS SDK
- Git
- PowerShell

后端 `pom.xml` 配置的 Java 版本是：

```xml

```

本机测试时 Java 23 也可以运行，但项目最低按 Java 17 理解。

### 1.2 推荐打开方式

根目录：

```powershell

```

手机端请在 DevEco Studio 中打开：

```text

```

不要只打开 `entry` 子目录。


## 2. 启动本地依赖

在根目录执行：

```powershell
.\scripts\dev-deps-up.ps1
```

该脚本实际执行：

```powershell
docker compose -f deploy/docker-compose.yml up -d redis minio
```

会启动：

| 服务 | 容器名 | 端口 | 用途 |
| --- | --- | --- | --- |
| Redis | `petpal-redis` | `6379` | 后端缓存/扩展依赖 |
| MinIO | `petpal-minio` | `9000` / `9001` | 社区图片上传和对象存储 |

默认账号：

| 服务 | 账号 | 密码 |
| --- | --- | --- |
| 本机 MySQL | `root` | `54321` |
| MinIO | `minioadmin` | `minioadmin` |

MinIO 控制台：

```text
http://127.0.0.1:9001
```

### 2.1 检查依赖是否启动

```powershell
docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"
```

至少应看到：

```text
petpal-redis
petpal-minio
```

### 2.2 数据库初始化说明

业务数据库不再由 `dev-deps-up.ps1` 启动。默认使用本机 MySQL：

```text
host: localhost
port: 3306
database: petpal
user: root
password: 54321
```

首次初始化时，先创建 `petpal` 数据库，再依次执行：

```text
petpal-server/src/main/resources/db/schema.sql
petpal-server/src/main/resources/db/seed.sql
```

如果 `petpal` 数据库已经存在，先检查表和数据，不要为了启动流程直接删除本机数据。

## 3. 启动后端

在根目录执行：

```powershell
.\scripts\run-backend.ps1
```

该脚本进入：

```text
D:\HongMengprogram\20260317\petpal-server
```

然后执行：

```powershell
mvn spring-boot:run
```

后端默认地址：

```text
http://127.0.0.1:18080
```

API 前缀：

```text
http://127.0.0.1:18080/api
```

### 3.1 后端配置位置

```text
petpal-server/src/main/resources/application.yml
```

关键配置：

```yaml
server:
  port: ${PETPAL_SERVER_PORT:18080}

spring:
  datasource:
    url: jdbc:mysql://localhost:3306/petpal?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai
    username: root
    password: 54321

petpal:
  admin:
    token: petpal-admin-token-change-me
  storage:
    endpoint: http://localhost:9000
    bucket: petpal
```

### 3.2 使用其他后端端口

如果要临时改后端 HTTP 端口，例如 `19080`：

```powershell
$env:PETPAL_SERVER_PORT='19080'
.\scripts\run-backend.ps1
```

注意：手机端和管理端也要同步改成同一个端口。

## 9. 常见错误与修复

### 9.1 后端端口 `8080` 启不起来

现象：

```text
Port 8080 was already in use
```

或 Windows 明明没有进程占用，但端口仍不能绑定。

原因：

- 本机可能保留了 `8000-8099` TCP 端口段。
- `8080` 可能落在系统 excluded port range 内。

检查：

```powershell
netsh interface ipv4 show excludedportrange protocol=tcp
```

修复：

- 使用默认 `18080`。
- 或临时指定其他端口：

```powershell
$env:PETPAL_SERVER_PORT='19080'
.\scripts\run-backend.ps1
```

同时修改：

- `Cutepetpost/entry/src/main/ets/config/PetPalAppConfig.ets`
- `petpal-admin` 页面里的 API Base URL

### 9.2 本机 MySQL `3306` 不可用

现象：

```text
Communications link failure
Access denied for user 'root'
Unknown database 'petpal'
```

原因：

- 本机 MySQL 没有启动。
- 账号密码不是 `root/54321`。
- `petpal` 数据库还没有用 `schema.sql` 和 `seed.sql` 初始化。

检查：

```powershell
netstat -ano | findstr :3306
```

修复：

1. 启动本机 MySQL，并确认监听 `localhost:3306`。
2. 确认 root 密码是 `54321`，或同步修改 `application.yml`。
3. 如果 `petpal` 不存在，创建数据库并执行初始化 SQL。

默认 datasource：

```yaml
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/petpal?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai
    username: root
    password: 54321
```

Docker MySQL 不再是默认开发路径。

### 9.3 后端连不上 MySQL

现象：

```text
Communications link failure
Access denied for user 'root'
Unknown database 'petpal'
```

检查本机 MySQL：

```powershell
netstat -ano | findstr :3306
```

检查数据库是否存在：

```powershell
mysql -uroot -p54321 -hlocalhost -P3306 -e "SHOW DATABASES;"
```

修复：

1. 确认本机 MySQL 正在运行。
2. 确认 `application.yml` 的账号密码是 `root/54321`。
3. 如果数据库初始化缺失，创建 `petpal` 并依次执行 `schema.sql`、`seed.sql`。

### 9.4 修改了 `schema.sql` 但数据库没变化

原因：

- `schema.sql` 和 `seed.sql` 需要手动应用到本机 MySQL。
- 已存在的 `petpal` 数据库不会自动重放初始化 SQL。

修复：

如果可以重建本地 `petpal` 数据库，先确认没有需要保留的数据，再执行重建和初始化 SQL。

如果不能删除数据，手动执行迁移 SQL：

```powershell
mysql -uroot -p54321 -hlocalhost -P3306 petpal
```

然后在 MySQL shell 中执行需要的 `ALTER TABLE` 或 `UPDATE`。

### 9.5 登录一直失败

现象：

- 手机端提示登录失败。
- 后端返回 `401`。
- 使用默认账号 `13800000001 / 123456` 也失败。

可能原因：

- 本地旧数据库里用户密码还是旧明文或旧 hash。
- 当前版本认证已是 BCrypt-only。

修复已知测试账号：

```sql
UPDATE user
SET password = '$2a$10$Trga2O1gLdnL6g8mvggVK.kdQdkQkkcEQ8AWzaS1oTD.t4eZLKfzi'
WHERE phone IN ('13800000001', '13800000002');
```

执行方式：

```powershell
docker exec -it petpal-mysql mysql -uroot -proot petpal
```

粘贴 SQL 后回车。

### 9.6 手机端访问不到后端

现象：

- DevEco 运行正常，但登录或列表加载失败。
- 后端本机浏览器能访问，手机端不行。

高概率原因：

- 手机端配置成了 `127.0.0.1`。
- 手机和电脑不在同一局域网。
- Windows 防火墙拦截。
- 后端端口和手机端配置不一致。

检查：

1. 后端是否启动：

```text
http://127.0.0.1:18080
```

2. 查电脑局域网 IP：

```powershell
ipconfig
```

3. 手机端配置是否类似：

```typescript
export const PETPAL_API_DEV_BASE_URL: string = 'http://192.168.x.x:18080/api';
```

修复：

- 不要使用 `127.0.0.1`。
- 不要漏掉 `/api`。
- 确认手机和电脑在同一个 Wi-Fi。
- Windows 防火墙允许 Java 或该端口入站。

### 9.7 管理端显示 `failed to fetch`

现象：

管理端页面打开了，但刷新时报：

```text
failed to fetch
```

检查：

- API Base URL 是否是 `http://127.0.0.1:18080`，不要带 `/api`。
- Admin Token 是否是 `petpal-admin-token-change-me`。
- 后端是否运行。
- 浏览器控制台 Network 是否有 CORS 或 OPTIONS 失败。

修复：

1. 后端和管理端端口保持一致。
2. 管理端配置：

```text
API Base URL: http://127.0.0.1:18080
Admin Token: petpal-admin-token-change-me
```

3. 如果命令行能调通但浏览器不行，优先检查 CORS/preflight。

### 9.8 MinIO 上传失败

现象：

```text
FILE_UPLOAD_FAILED
FILE_REQUIRED
INVALID_FILE_TYPE
FILE_TOO_LARGE
```

规则：

- 上传字段名必须是 `file`。
- 只允许图片。
- 单文件最大 5MB。
- 请求最大 6MB。
- 图片 URL 应通过后端代理 `/api/file/object/{fileKey}`，手机端不要依赖 MinIO 私有直链。

检查：

```powershell
docker ps --filter "name=petpal-minio"
docker logs petpal-minio --tail 100
```

打开：

```text
http://127.0.0.1:9001
```

确认 MinIO 可登录。

### 9.9 HAP 构建提示 signingConfigs

现象：

```text
Will skip sign 'hos_hap'. No signingConfigs profile is configured
```

说明：

- 本地 debug 构建可以通过。
- 这是签名配置缺失警告，不是编译失败。

如果只是本地验证，看到：

```text
BUILD SUCCESSFUL
```

即可继续。

如果要安装到真机，需要在 DevEco Studio 中配置签名。

### 9.10 ArkTS warning：函数可能抛异常

现象：

```text
ArkTS:WARN File: ... Index.ets
Function may throw exceptions. Special handling is required.
```

当前已知位置来自图片上传时的文件读取逻辑。

说明：

- 当前构建不被阻断。
- 如果未来严格模式升级为 error，需要给对应 `fs.openSync`、`fs.statSync`、`fs.readSync`、`fs.closeSync` 加更细粒度异常处理。

### 9.11 Hypium 测试显示 `BUILD SUCCESSFUL` 但下面有 `ERROR`

现象：

```text
BUILD SUCCESSFUL
ERROR: Error in ...
```

说明：

- hvigor 可能仍返回构建成功。
- 但 Hypium 断言失败仍然代表测试失败。

处理：

- 搜索日志里的 `ERROR: Error in`。
- 先修第一个失败用例。
- 不要只看 `BUILD SUCCESSFUL`。

### 9.12 PowerShell 执行 npm 或脚本被拦截

现象：

```text
running scripts is disabled on this system
```

修复：

- 优先使用项目已有 `.ps1` 脚本。
- 如果是 npm/npx，使用 `npm.cmd` 或 `npx.cmd`。
- 不建议为了一个命令全局放开执行策略。

### 9.13 Docker Desktop 没启动

现象：

```text
Cannot connect to the Docker daemon
```

修复：

1. 启动 Docker Desktop。
2. 等待左下角显示 Docker Engine running。
3. 重新执行：

```powershell
.\scripts\dev-deps-up.ps1
```

### 9.14 后端接口 401 或 403

常见原因：

- 手机端 access token 过期或无效。
- 把 refresh token 当 access token 使用。
- 管理端缺少 `X-PetPal-Admin-Token`。

修复：

- 手机端重新登录。
- 管理端填写 `petpal-admin-token-change-me`。
- 如果数据库用户密码是旧数据，参考 9.5 修复。

### 9.15 Community feed 或详情不应该显示 mock 数据

当前规则：

- Community P0 不允许后端失败时静默回退 mock 数据。
- `/post/feed` 失败时应显示错误或空状态。
- 不存在或已删除的 post id 应显示 not found，不能显示第一条缓存帖子。

如果再次出现 mock 数据：

1. 检查 `PetPalRepository.ets`：

```text
postCache` 初始值应为空数组
getPostById()` 找不到应返回 null
getComments()` 未命中应返回空数组
```

2. 跑手机端 repository 测试：

```powershell
cd D:\HongMengprogram\20260317\Cutepetpost
# 执行 5.4 的 hvigorw.js test 命令
```

### 9.16 中文显示乱码

规则：

- 源码、文档、验收记录保持 UTF-8。
- 不要用错误编码重写 ArkTS 或 Markdown 文件。

修复建议：

- 使用 DevEco Studio、VS Code 或支持 UTF-8 的编辑器。
- 不要用老版命令行工具批量改中文。
- 如果只是一处文案乱码，只修当前任务相关文案，不做全文件无关清理。

```powershell
docker compose -f deploy/docker-compose.yml down
```

停止并删除本地 MySQL/MinIO 数据卷：

```powershell
docker compose -f deploy/docker-compose.yml down
docker volume rm deploy_mysql-data
docker volume rm deploy_minio-data
```

注意：删除 volume 会清空本地数据。

## 12. 推荐排错顺序

遇到问题时按这个顺序查，不要一开始就改代码：

1. `git status --short --branch`：确认当前分支和是否有未提交改动。
2. `docker ps`：确认 MySQL、Redis、MinIO 是否运行。
3. `.\scripts\test-backend.ps1`：确认后端行为和数据库测试是否通过。
4. `.\scripts\run-backend.ps1`：看后端启动日志里的第一个 ERROR。
5. 浏览器或 PowerShell 调后端登录接口。
6. 检查手机端 `PETPAL_API_DEV_BASE_URL` 的 IP 和端口。
7. 检查管理端 base URL 和 admin token。
8. 再看 DevEco/HAP 构建日志。
9. 最后才定位业务代码。
