# Post-MVP Quality Cleanup Plan

> 计划日期：2026-04-22  
> 状态：done  
> 目标分支：`main`

## Owner

- Codex

## Entry Criteria

- `main` 已合并刚刚的小修复。
- `git status --short` 仍然干净。
- 现有 `docs/phone-mvp.md` 和审查文件已经读过。

## 背景

前一轮审查已经确认，`main` 的 MVP 主链路可用，刚刚的小修复也已经把一批真实问题收掉了：

- 预约详情查询不再走全表内存过滤。
- 社区 feed 已加入可选分页参数，并把图片和 `liked` 状态改成批量加载。
- 登录页去掉了重复的 session 持久化。
- `PetPalRepository.ets` 删除了几个无人调用的空同步方法。
- 数据库密码改成了环境变量可覆盖的写法。

这份计划只处理下一步该收口的质量问题，不扩新功能，不改产品方向。

## 目标

1. 降低手机端数据层的维护成本。
2. 去掉明显的遗留壳和外部不稳定依赖。
3. 保持已验收的电话端主流程不回退。

## 本轮范围

### 1. 统一手机端请求入口

- 保留 `PetPalRepository` 作为页面层的稳定入口，不改页面调用方式。
- 把重复的请求封装逐步收拢到单一层，优先复用 `PetPalHttpClient.ets` / `PetPalApiClient.ets` 现有能力。
- 先覆盖稳定的读写路径：登录、用户资料、宠物、机构、预约、社区读写。
- Multipart 上传先按当前可用实现保留，不为了“统一”硬拆。

### 2. 清理遗留空壳和外部依赖

- 确认 `DemoDataStore.java` 没有运行时引用后删除；如果暂时不能删，就只保留一个明确标记，不再让它参与核心路径。
- 把后端默认宠物头像从外部随机服务收敛到稳定来源，避免新建宠物依赖外部站点可用性。
- `seed.sql`、`MockPetPalData.ets` 里的历史外部图片可以保留到后续单独清理，不和本轮主目标混在一起。

### 3. 再做文件级收口

- 等数据层稳定后，再考虑把 `PetPalRepository.ets` 按领域继续拆小。
- `Index.ets` 体积过大属于下一档整理项，不和这轮数据层收口一起做。
- 这一步只做结构整理，不改页面行为和接口形状。

## 实施顺序

1. 先收敛手机端请求入口，确保现有测试还能过。
2. 再清理 `DemoDataStore` 和默认头像依赖。
3. 最后决定是否开始拆 `PetPalRepository` / `Index.ets`。

## 验收门槛

- 后端测试通过。
- `Cutepetpost` 的 `hvigor test` 通过。
- `Cutepetpost` 的 `assembleHap` 通过。
- 手机端现有 repository / service 测试不回退。
- 手工验证至少覆盖登录、预约浏览、社区发帖/图片、宠物档案主流程。

## 明确不做

- 不做 Refresh Token 自动刷新。
- 不做搜索、地图、通知、评论二级回复。
- 不做平板/手表适配。
- 不做一次性重写 `PetPalRepository.ets` 或 `Index.ets`。

## 风险

- 后端和手机端仍有历史图片 URL、seed 数据和 mock 数据残留，需要分批清理，不能一次全删。
- `PetPalApiClient.ets` 目前还有测试引用，不能因为“看起来重复”就直接删。
- 默认头像来源替换后，需要同步检查宠物创建、宠物列表和相关测试断言。

## 退出标准

本计划完成后，仓库应该满足下面三点：

- 已知的空壳代码不再参与主流程。
- 手机端请求链路只保留一套清晰的主路径。
- 现有 MVP 验证和小修复后的测试结果保持稳定。
